Slide 3: Model Client-Server
poin di materi:
Arsitektur jaringan untuk pertukaran data melalui internet.


Client = pihak yang meminta data atau layanan.
Contoh: Browser (Chrome, Firefox), smartphone, aplikasi mobile.
Tugas: Mengirim request, menampilkan hasil, menyediakan antarmuka pengguna (UI).
Server = pihak yang menyediakan data atau layanan.
Tugas: Menerima request, memproses data, mengirim response ke client.
Selalu aktif mendengarkan permintaan melalui port tertentu.
Analogi Restoran:
Pelanggan = Client (memesan makanan)
Dapur = Server (menyiapkan makanan)
Fakta Penting:
Server tidak harus berupa satu komputer fisik.
Dapat berupa beberapa komputer, virtual machine, atau cloud server.
Server dapat mengakses data dari database atau layanan lain.
Alur Sederhana:Client → Request → Server → Response → Client


slide 4: Cara Kerja Model Client-Server
Request (Permintaan)
Client mengirim permintaan data atau layanan ke server.
Pemrosesan di Server
Server menerima dan memproses permintaan.
Dapat mengambil data dari database atau layanan lain.
Response (Tanggapan)
Server mengirim hasil pemrosesan kembali ke client.
Render User Interface (UI)
Client menampilkan data kepada pengguna dalam bentuk teks, gambar, tombol, tabel, atau grafik.
Alur Komunikasi:
Client → Request → Server → Processing → Response → Client → UI Display


slide 5: API sebagai Jembatan Komunikasi
API (Application Programming Interface)
API adalah penghubung komunikasi antara client dan server.
Berfungsi sebagai kontrak atau aturan komunikasi yang disepakati bersama.
Client mengirim permintaan melalui API.
Server memproses permintaan dan mengembalikan data melalui API.
Keuntungan API:
Client tidak perlu mengetahui cara kerja internal server.
Menyederhanakan pertukaran data dan layanan.
Memungkinkan integrasi antar aplikasi yang berbeda.
Analogi: API = Menu restoran yang digunakan pelanggan untuk memesan tanpa perlu masuk ke dapur.


slide 6: HTTP sebagai Bahasa Komunikasi
Dalam komunikasi client dan server, protokol yang paling umum digunakan adalah HTTP (Hypertext Transfer Protocol).


HTTP berfungsi sebagai aturan yang mengatur bagaimana data dikirim dan diterima melalui internet.


Contoh sederhana:
Browser → HTTP Request → Server
Server → HTTP Response → Browser


Tanpa HTTP atau protokol jaringan lainnya, client dan server tidak dapat saling berkomunikasi.

slide 9: Apa itu API?
API (Application Programming Interface) adalah sekumpulan aturan dan mekanisme yang memungkinkan dua aplikasi atau sistem perangkat lunak untuk saling berkomunikasi.

API menentukan:

Permintaan apa yang dapat dilakukan.
Data apa yang harus dikirim.
Format data yang digunakan.
Respons yang akan diberikan.

Dengan kata lain, API bertindak sebagai perantara komunikasi antara client dan server.

slide 10: Analogi Sederhana
API sebagai Menu Restoran
Bayangkan Anda berada di sebuah restoran.

Anda adalah client.
Dapur adalah server.
Menu restoran adalah API.
Pelanggan tidak perlu masuk ke dapur dan memahami cara memasak makanan.

Pelanggan cukup melihat menu yang tersedia lalu memesan makanan sesuai aturan yang ada.

Begitu pula pada API.

Client tidak perlu mengetahui bagaimana kode program server dibuat. Client hanya perlu mengetahui:

Permintaan apa yang tersedia.
Data apa yang harus dikirim.
Hasil apa yang akan diterima.

slide 13-14-15: Komponen Utama dalam API

Ketika membaca dokumentasi API, terdapat beberapa komponen penting yang harus dipahami.

1. Endpoint
Endpoint adalah alamat tujuan API yang akan dipanggil oleh client.

Contoh:

https://api.website.com/users
Alamat tersebut menunjukkan lokasi sumber data yang ingin diakses.

Analogi
Jika API adalah restoran, maka endpoint adalah meja atau loket tertentu tempat Anda melakukan pemesanan.

2. HTTP Method
Method menentukan tindakan yang ingin dilakukan terhadap data.

Method yang paling sering digunakan adalah:

Method	Fungsi
GET	Mengambil data
POST	Menambahkan data baru
PUT	Memperbarui seluruh data
PATCH	Memperbarui sebagian data
DELETE	Menghapus data
Contoh
Mengambil daftar pengguna:

GET /users
Menambahkan pengguna baru:

POST /users
3. Parameter
Parameter adalah data yang dikirim client agar server dapat memproses permintaan.

Required Parameter
Wajib dikirim.

Jika tidak ada, server akan menolak permintaan.

Contoh:

{
  "username": "Purwa"
}
Optional Parameter
Boleh dikirim atau tidak.

Biasanya digunakan untuk fitur tambahan.

Contoh:

{
  "username": "Purwa",
  "location": "Bandung"
}
4. Authentication
Authentication digunakan untuk memastikan identitas pengguna yang memanggil API.

Tujuannya adalah:

Mencegah penyalahgunaan.
Menjaga keamanan data.
Mengontrol akses pengguna.
Contoh:

Authorization: Bearer abc123xyz
Token tersebut berfungsi seperti kartu identitas ketika mengakses layanan API.

5. Rate Limiting
Rate Limiting adalah pembatasan jumlah request yang boleh dilakukan dalam periode tertentu.

Contoh:

100 request per jam
Jika batas tersebut terlampaui, server dapat menolak request berikutnya.

Tujuannya untuk:

Mengurangi spam.
Mengurangi beban server.
Mencegah penyalahgunaan layanan.
6. Response Format
Response adalah data yang dikembalikan oleh server.

Format yang paling umum digunakan saat ini adalah JSON (JavaScript Object Notation).

Contoh:

{
  "id": 1,
  "nama": "Purwa",
  "jurusan": "Informatika"
}
JSON populer karena:

Ringan.
Mudah dibaca manusia.
Mudah diproses oleh program.

slide 16-17: Alur Kerja API
Secara umum proses komunikasi API berlangsung sebagai berikut:

1. Client Mengirim Request
GET /users/1
2. API Menerima Request
API memeriksa:

Endpoint
Method
Parameter
Authentication
3. Server Memproses Request
Server dapat:

Mengambil data dari database.
Menyimpan data baru.

Berkomunikasi dengan layanan lain.

Contoh:

  "id": 1,

}

Data yang diterima kemudian ditampilkan kepada pengguna dalam bentuk:

Teks
Tabel
Tombol
Grafik
Halaman web

slide 18:Studi Kasus: Mengirim Tweet melalui API
Misalkan sebuah aplikasi ingin membuat tweet secara otomatis.

Request
Client mengirimkan:

POST /statuses/update
Dengan data:

{
  "status": "Hello World"
}
Proses di Server
Server akan:

Memeriksa identitas pengguna.
Memastikan teks tweet valid.
Memastikan tweet tidak duplikat.
Menyimpan tweet ke database.
Response Sukses
{
  "id": 123456,
  "text": "Hello World",
  "created_at": "2025-05-24"
}
Response Gagal
Jika pengguna mencoba mengirim tweet yang sama secara berulang:

{
  "error": "Status is a duplicate"
}
Server juga dapat mengirim kode error seperti:

403 Forbidden
yang menandakan bahwa permintaan ditolak.
